# AI-DLC (AI-Driven Development Lifecycle)

## 한 줄 요약

AWS가 2025년 re:Invent에서 발표한 소프트웨어 개발 방법론. SDLC(소프트웨어 개발 생명주기)를 AI 중심으로 재설계한 것.

---

## 왜 나왔나

AI 코딩 어시스턴트(GitHub Copilot, Amazon Q 등)를 도입했는데 생산성이 오히려 떨어지는 팀이 많았다. 이유는 도구 문제가 아니라 **방식의 문제**였다.

기존 SDLC에 AI를 도구로 끼워 넣는 방식의 한계:
- 요구사항은 사람이 분석하고, AI는 코드만 생성
- 각 단계가 분리돼 있어 AI가 전체 맥락을 모름
- 결과적으로 AI가 컨텍스트 없이 코드 조각만 만들어냄

AI-DLC는 **AI를 도구가 아니라 개발 프로세스의 중심 협업자**로 재정의한다. 기획부터 운영까지 AI가 일관된 맥락을 가지고 각 단계에 참여한다.

---

## 세 가지 단계

### 1. Inception (기획/요구사항)

비즈니스 요구사항을 AI가 사용자 스토리, 인수 조건, 태스크 단위로 변환한다.

핵심 프랙티스: **Mob Elaboration**
팀 전체가 AI의 질문과 제안을 같이 검토하고 검증하는 방식이다. AI가 요구사항을 분석하면서 모호한 부분을 질문으로 던지고, 팀이 함께 답한다. 개인이 스펙을 쓰는 게 아니라 팀이 AI와 함께 스펙을 정제한다.

Brownfield(기존 코드베이스) 프로젝트는 이 단계에서 **역공학(Reverse Engineering)** 을 먼저 수행한다. 기존 코드를 그대로 AI에게 던지는 게 아니라, 의미 있는 컨텍스트(Semantic Context)로 정제해서 다음 단계로 넘긴다.

### 2. Construction (설계/구현)

Inception에서 만든 컨텍스트를 바탕으로 AI가 아키텍처, 도메인 모델, 코드, 테스트를 제안한다.

핵심 프랙티스: **Mob Construction**
팀이 실시간으로 기술 결정과 아키텍처 선택에 개입한다. AI가 제안하면 팀이 피드백하고, 그 피드백이 다시 컨텍스트로 쌓인다. 단계가 진행될수록 AI가 가진 컨텍스트가 풍부해진다.

### 3. Operations (운영)

앞 두 단계에서 쌓인 컨텍스트를 AI가 인프라 코드, 배포, 모니터링에 활용한다. 운영 단계에서도 개발 초기의 설계 의도와 요구사항을 AI가 이해한 상태로 작업한다.

---

## SDLC vs AI-DLC 차이

기존 SDLC는 단계가 순차적이고 AI는 각 단계에서 독립적인 도구로 쓰인다.

```
요구사항 분석 → 설계 → 구현 → 테스트 → 배포
    (사람)     (사람)  (AI 보조) (사람)  (사람)
```

AI-DLC는 AI가 전체 흐름에 걸쳐 맥락을 유지한다.

```
Inception → Construction → Operations
  (AI + 팀)   (AI + 팀)    (AI + 팀)
      ↑____________↑____________↑
          컨텍스트가 단계마다 누적됨
```

핵심 차이는 **컨텍스트의 연속성**이다. 요구사항 단계에서 논의한 내용이 설계에, 설계에서 결정한 내용이 운영에 이어진다.

## 실제 성과 (AWS 발표 기준)

100개 이상의 고객 실험 기반으로 발표된 수치들:
- Wipro: 3개월 분량의 작업을 20시간에 완료
- Dun: 48시간 만에 신규 애플리케이션 구축 후 다음 주 런칭
- 일반적으로 3~10x 생산성 향상 보고

단, 속도만이 목표가 아니라 품질과 예측 가능성을 동시에 달성하는 게 방법론의 목표라고 강조한다.

---

## Kiro와의 관계

Kiro IDE(AWS가 2025년 출시한 에이전트 IDE)는 AI-DLC 방법론을 실제로 구현한 도구다.

- Kiro의 Spec 파일(requirements.md, design.md, tasks.md) = AI-DLC의 Inception 단계 산출물
- Kiro의 Steering Files = 단계 간 컨텍스트 유지 수단
- Kiro의 Agent Hooks = Construction/Operations 단계 자동화

방법론(AI-DLC)과 도구(Kiro)를 같이 이해하면 AWS가 그리는 전체 그림이 보인다.

참고: kiro_ide.md

## 요약

- SDLC에 AI를 도구로 끼우는 게 아니라, 프로세스 자체를 AI 중심으로 재설계한 방법론
- Inception → Construction → Operations, 단계마다 컨텍스트가 누적됨
- Mob Elaboration, Mob Construction — 팀이 AI와 함께 각 단계를 진행
- Brownfield 프로젝트는 역공학으로 시맨틱 컨텍스트를 먼저 만들고 시작
- Kiro IDE가 이 방법론의 실제 구현체
