**배경**: `backend/src/index.ts`가 155줄, 라우트 9개가 순서 없이 나열되어 있어 라우팅 레이어만 도메인별로 분리.

**구조 결정 과정**

- 처음 제안한 6개 라우트 파일(`auth`/`user`/`admin`/`quest`/`session`/`ws`) 분리안을 사용자가 그대로 채택 — "찾기 편하니까"가 이유. 파일당 코드량이 적어도 이름으로 바로 찾을 수 있는 이점을 우선.
- 라우트 파일명이 서비스 파일명과 겹치는 문제(`auth.ts` vs `routes/auth.ts`) 지적받아 `*.routes.ts` 접미사로 전체 통일 (`terminal.ts`도 `ws.ts` 대신 `terminal.routes.ts`로).
- 서비스 파일(`auth.ts`, `quest.ts`, `user.ts`, `sandbox.ts`, `terminal.ts`)도 `services/`로 이동 — "파일 이동일 뿐이니 간단하다"는 사용자 판단으로 같은 라운드에 포함.

**구현 완료 후 검증**

- `git mv`로 서비스 파일 이동 확인 (rename으로 인식됨)
- `tsc --noEmit` 통과
- `admin.routes.ts`에 남아있던 기존 버그(`PATCH /admin/users/:id/password`가 `request.params`에서 `password`를 읽어 항상 undefined) 발견 → 수정
- `services/user.ts`와 `routes/user.routes.ts`의 내용 불일치 발견 (진행/리더보드만 있고 계정 관리는 admin으로 분리되어 있음) — 이름과 실제 내용이 어긋나는 문제로 남아있음, 추후 재고 필요

**최종 구조**

```
backend/src/
  db.ts, types.ts, index.ts
  services/  auth.ts, quest.ts, user.ts, sandbox.ts, terminal.ts
  plugins/   docker.ts, auth-guard.ts
  routes/    auth.routes.ts, user.routes.ts, admin.routes.ts,
             quest.routes.ts, session.routes.ts, terminal.routes.ts
```

### Phase 7 접근 제어 설계 논의 → grilling으로 명세 확정

업무 관련 퀘스트 세트(사내 장애/작업 사례)는 특정 유저에게만 공개해야 한다는 요구사항 논의.
`/grill-with-docs`로 관통 인터뷰 진행, `docs/specs/spec_phase7_quest_access.md` 신규 작성 + 확정.

**확정된 설계**

- 접근 제어 단위: quest_set (퀘스트 단위 아님), 기본 공개 + 예외 비공개
- DB: `quest_set.is_public` 플래그 + `quest_set_access`(user_id, quest_set_id) 매핑 테이블
- `GET /quest-sets` 인증 필수로 전환 — 프론트는 이미 토큰을 보내고 있어(무시되고 있었을 뿐) 프론트 변경 불필요
- admin은 `quest_set_access` 등록 여부와 무관하게 role만으로 전체 세트 접근 (관리 목적)
- `quest_attempt`, `getProgress`/`getLeaderboard` 쿼리는 변경하지 않음 — 접근 제어는 "지금 볼 수 있는 목록"에만 적용, 할당 해제돼도 과거 이력은 Progress/Leaderboard에 계속 표시
- `is_public` 토글은 `quest_set_access`에 영향 없음 — 재비공개 시 예전 할당이 자동 복원됨, 최소 할당 인원 제약 없음
- `GET /admin/users` API 신규 필요 (현재 부재 확인) — 접근 할당 UI에서 유저 선택용
- 관리자 화면은 URL 라우터 도입 없이 기존 `App.tsx`의 `page` state 패턴에 `'admin'` 추가, SideNav에 admin role 조건부 메뉴, member가 진입 시 프론트에서 home으로 리다이렉트(UX 목적, 실제 보안 경계는 `adminMiddleware`)
- 유저 선택 UI는 검색 없는 체크박스 전체 나열 (현재 유저 규모에서 충분)

**진행 방식 결정**: 명세만 이 세션에서 확정하고, 구현 가이드 작성 + 실제 구현은 다음 세션으로 넘김 — 이미 이번 세션에서 7g/7f 구현+검증을 마쳤고, DB 마이그레이션 + 다중 라우트 + 신규 관리자 화면까지 한 세션에 얹으면 검증이 부실해질 위험 판단.
