**검증용 계정 준비**

- 기존 시드는 admin 1 + member 1(`test@okestro.com`)뿐이라 "할당 O vs 할당 X" 유저를 동시 비교할 수 없는 문제 → `test2@okestro.com` member 계정 추가
- 비밀번호를 처음엔 `password123`으로 잘못 판단해 해시 생성 — 사용자가 "cloud1234!" 아니냐고 지적, bcrypt.compareSync로 실제 해시 검증해 기존 계정 비밀번호가 `cloud1234!`였음을 확인 후 test2도 동일하게 통일. **교훈: 계정 비밀번호를 가정하지 말고 사용자에게 먼저 확인하거나 실제 해시로 검증할 것.**

**curl 검증 중 발견한 버그 3건**

1. `quest.routes.ts`의 `GET /quest-sets`에 `authMiddleware` 누락 — `req.user` undefined로 500. 사용자가 가이드를 보고 직접 구현하다 빠뜨림, AI가 원인만 설명 → 사용자가 "수정 허가" 발언 후 AI가 직접 수정
2. `services/quest.ts`의 `getQuestSetsForAdmin`에서 SQL 오타 — `ON u.id = qsa_user_id`(언더스코어, 존재하지 않는 컬럼) → `qsa.user_id`로 수정. SELECT 별칭(`u.id AS userId`)과 매핑 코드(`a.user_id`)의 불일치도 함께 수정
3. `tsx src/index.ts`가 watch 모드가 아니라 파일 수정이 반영 안 되는 문제 반복 발생 — 매 수정 후 프로세스를 kill하고 `npm run dev`로 재시작해야 했음

**브라우저 검증 중 발견한 버그 2건**

4. `Admin.tsx`가 `SideNav`에 `userRole`/`onAdmin`을 전달하지 않아, 관리자 화면 진입 후 사이드바에서 "관리자" 메뉴가 사라짐 — `App.tsx`의 `<Admin>` 호출부도 `userRole` 누락되어 있었음, 둘 다 수정
5. 공개/비공개 토글·유저 할당 버튼이 반응 없음 — 콘솔 확인 결과 `Access-Control-Allow-Methods in preflight response`로 PATCH 차단. `backend/src/index.ts`의 `fastify.register(cors, { origin: true })`에 `methods` 옵션이 없었던 게 원인 → `methods: ['GET','POST','PATCH','DELETE','OPTIONS']` 명시로 해결

**curl 시나리오 → 브라우저 체크리스트로 교체**

- 가이드 초안엔 curl 기반 백엔드 검증 시나리오를 상세히 작성했으나, 사용자가 화면으로 직접 테스트하길 원해 "curl은 AI가 하고, 화면 테스트는 사용자가 한다"로 역할 분담
- 가이드의 "검증" 섹션을 curl 스크립트 대신 13단계 브라우저 체크리스트로 재작성

**UX 개선 — 접근 유저 체크박스 아코디언화**

- 세트가 많아지면(14개) 접근 유저 섹션이 테이블 맨 아래에 나타나 스크롤을 한참 내려야 하는 문제 지적받음
- 대안 3가지(아코디언/테이블 위 이동/사이드패널) 중 아코디언 방식 추천·채택 — 해당 세트 행 클릭 시 바로 아래(`<tr colSpan>`)에 펼쳐지는 구조로 변경, 재클릭 시 접힘
- 유저 수가 늘어날 걸 대비해 체크박스 목록에 `max-h-64 overflow-y-auto` 추가
- 가이드와 실제 `Admin.tsx` 모두에 반영 (Fragment key 이슈 있어 `<>` 대신 `<Fragment key={s.id}>` 사용)

**접근 유저 목록에서 admin 제외**

- 체크박스 목록에 admin 계정도 섞여 나오는 게 이상하다는 지적 → admin은 role 우회로 이미 모든 세트에 접근 가능해 개별 할당 대상이 아님이 맞다고 판단
- `services/user.ts`의 `getAllUsers`에 `WHERE role = 'member'` 조건 추가, 가이드에도 반영

**진행 방식 결정**: Phase 7h 구현/검증 완료. 세션 종료 시점에 사용자가 서버 프로세스를 직접 컨트롤하겠다고 해 AI가 백엔드를 종료함.

- `compartment_ocid`는 별도 compartment 없이 root compartment 사용 — root compartment OCID가 tenancy_ocid와 동일하다는 점을 사용자가 헷갈려해서 설명

**ubuntu_image_ocid를 콘솔 수동 조회 대신 자동화로 전환**

- 콘솔에서 Ubuntu 22.04 ARM64 이미지 OCID를 못 찾겠다는 사용자 요청 → `main.tf`에 `oci_core_images` data source를 추가해 리전 무관 자동 조회로 변경(사용자가 "허가"). `ubuntu_image_ocid` 변수/tfvars 항목 제거
- 가이드에도 동일 반영

**파일 오타 수정**

- `infra/terraform/variables..tf`(점 두 개) → `variables.tf`로 정정
- `.gitignore`에 Terraform 관련 항목이 중복 기재되어 있던 것 정리

**terraform init/apply 트러블슈팅**

1. HCL 문법 오류 — `tcp_options { min = 22; max = 22 }`처럼 세미콜론으로 구분한 게 원인. HCL은 세미콜론 미지원, 줄바꿈으로 변경 (main.tf + 가이드 둘 다)
2. 401 Not Authenticated — 처음엔 `private_key_path`의 `~` 미확장을 의심해 절대경로로 변경했으나 원인 아니었음. 실제 원인은 `user_ocid` 오타(`voczhv` vs `vozhv`, 사용자가 화면 캡처를 옮겨적으며 발생) — 콘솔 My profile에서 재확인 후 수정, 인증 통과
3. **ARM Always Free 용량 부족 — "Out of host capacity" (500 에러)**. 네트워크 리소스(VCN/subnet/IGW/route table/security list)까지는 생성 성공, VM 생성 단계에서 반복 실패
   - 대응: x86 Always Free(E2.1.Micro)로 임시 전환하는 대안도 검토했으나, k8s/도커 샌드박스 다수 운영에 24GB RAM이 실제로 필요하다는 이유로 ARM 계속 대기 쪽으로 결정
   - `infra/scripts/retry_apply.sh` 신규 작성 — 지정 간격(기본 60초)으로 `terraform apply -auto-approve` 반복, 성공 시 자동 종료. 60초 간격이 OCI API rate limit에 안전한 수준임을 확인
   - 세션 종료 시점까지 계속 재시도 중 (미완료)

### Phase 10 명세 신설 — vcluster 전환 + KLID CMP 현장실습

**배경**: `docs/sources/KLID_CMP_deploy_guide.md`(오케스트로 CMP 배포 절차, 여러 namespace + Helm + ClusterRole/CRD 다룸)를 현장실습형 퀘스트 세트로 만들고 싶다는 요청.

**핵심 판단**: 이 실습은 `k8s_cluster_isolation.md`가 정리한 "namespace로 격리 안 되는 것"(ClusterRole, CRD, 클러스터 레벨 리소스, `get nodes`)과 정확히 겹쳐, 기존 Phase 6(namespace 격리) 방식으로는 동시 실습 자체가 불가능 — vcluster(완전 격리) 전환이 선결 조건이라는 결론.

**현장실습형 vs 트러블슈팅형 논의**

- 처음엔 "현장실습형(가이드 그대로 재현)이 이미 있는 절차라 더 쉽다"고 판단했으나, 사용자가 "트러블슈팅형이 범위가 더 좁지 않냐"고 반박 — 실제로 트러블슈팅형은 정상 배포 위에서 한 가지 고장만 재현하면 되니 스코프가 더 작을 수 있음을 인정
- 다만 사용자가 "지금은 현장실습형이 급하다"고 최종 결정, 트러블슈팅형은 vcluster 인프라가 자리잡은 뒤 같은 환경 위에 추가하는 것으로 범위 밖 처리

**명세 구성**

- Phase 번호는 Phase 9(퀘스트 콘텐츠 확장 2, 예약됨) 다음인 Phase 10으로 결정. vcluster 전환과 KLID CMP 퀘스트를 하나의 Phase로 통합(vcluster는 이 실습만을 위한 선행조건이라는 이유)
