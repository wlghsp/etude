# AI 효율 활용법 전파 — 준비 로드맵

## 목표

"효율적인 AI 사용이란 무엇인가"를 먼저 정의하고, 데이터로 검증한 뒤, 그 결과를 발표한다.
결론을 미리 정해놓고 진행하는 게 아니라 연구에서 결론이 나오는 구조로 간다.

---

## Phase 0 — 효율적 AI 사용의 정의 (1주)

발표 준비보다 이게 먼저다. 정의 없이 데이터를 모으면 무엇을 찾아야 하는지 모른 채 진행하게 된다.

### 질문

- 효율적인 AI 사용이란 무엇인가?
- 토큰 절약만이 효율인가, 아니면 결과 품질도 포함되는가?
- 빠르게 답을 얻는 것과 정확한 답을 얻는 것 중 무엇이 우선인가?
- AI 의존도가 낮은 게 효율적인 건가, 아니면 AI를 많이 쓰더라도 결과가 좋으면 효율적인 건가?

### 가설 수립

위 질문에 대한 본인 답을 먼저 정리한다.
이게 데이터 수집과 분석의 기준이 된다.
데이터를 보고 가설이 맞는지 틀린지를 확인하는 것이 연구다.

### 핵심 관점 — 도메인 지식이 토큰을 줄인다

토큰 절약을 "어떻게 프롬프트를 잘 쓸까"로 접근하면 프롬프트 기술에 의존하게 된다.
다르게 보면: 내가 많이 알수록 AI에게 덜 물어도 된다.

모르는 사람 — "이 에러 어떻게 고쳐?" → AI가 문제 파악부터 시작, 여러 번 주고받음
아는 사람 — "NullPointerException, 이 라인에서 Optional 처리 빠진 것 같은데 맞나?" → 확인만, 한 번에 끝

같은 문제인데 대화 횟수와 토큰 소모량이 몇 배 차이 난다.

이것이 "AI를 잘 쓰는 것과 도메인을 잘 아는 것이 분리되지 않는다"는 이유다.
Anthropic 연구에서도 같은 결론이 나왔다 — AI를 쓰면서도 개념 이해를 유지한 그룹이 가장 높은 점수를 받았다.

참고: research_ai_skill_formation.md

## 발표 핵심 메시지

AI를 잘 쓰는 사람은 매번 잘 물어보는 게 아니라, 다시 물어볼 필요가 없는 구조를 만든다.

- 설정 파일, 로그, 가이드 등 실제 컨텍스트를 먼저 구성해서 AI가 환경을 파악하게 한다
- 그 컨텍스트와 결과를 문서로 남겨서 재사용한다
- 쌓일수록 AI와의 대화 비용이 낮아지는 구조가 된다

---

## AI 활용 방식 — 지호님 케이스

AI에게 서버에 직접 접근하거나 실행을 시키지 않는다.
실제 서버의 설정 파일, 설치 가이드, 로그 등을 한데 모아서 제공하고, AI는 이를 기반으로 분석과 가능성 좁히기만 담당한다. 실제 해결은 본인이 직접 한다.

이 방식의 강점:

- 실수가 적다 — AI가 직접 실행하다 잘못되는 상황 자체가 없다
- 본인 이해도가 유지된다 — 같은 문제가 다시 나왔을 때 AI 없이도 처리할 수 있다
- 토큰 효율이 좋다 — 컨텍스트가 정확하면 AI가 한 번에 핵심을 짚어줘서 추가 대화가 줄어든다
- 재사용 구조가 생긴다 — 컨텍스트와 결과를 문서로 남기면 다음번엔 처음부터 다시 설명할 필요가 없다

팀원들 중 AI한테 "이거 고쳐줘", "이거 실행해줘"로 쓰는 사람이 많을수록 이 방식의 차이가 발표에서 더 와닿는다.

## Phase 1 — AI 동작 원리 이해 (1~2주)

발표에서 "왜 이렇게 해야 하는가"를 설명하려면 원리 이해가 먼저다.
팁만 나열하면 팀원들이 외우다 잊는다. 원리를 알면 스스로 응용한다.

### 공부할 개념

토큰이란 무엇인가
- 글자 단위가 아닌 이유
- 입력 토큰 + 출력 토큰이 비용/품질에 어떻게 영향을 주는가
- Cursor에서 토큰이 어떻게 소모되는가 (코드 컨텍스트 자동 포함 구조)

컨텍스트 윈도우
- 대화가 길어질수록 왜 답이 이상해지는가
- 언제 새 대화(New Chat)를 시작해야 하는가
- Cursor에서 파일을 @ 로 첨부할 때 무슨 일이 생기는가

확률적 생성
- 왜 같은 질문에 다른 답이 나오는가
- 모호한 질문이 왜 엉뚱한 답을 내는가
- 구체적인 조건을 주면 왜 결과가 달라지는가

시스템 프롬프트 / 역할 부여
- Cursor Rules가 실제로 어떻게 동작하는가
- 역할을 명시하면 왜 일관된 답이 나오는가

### 학습 방법

각 개념을 Claude한테 직접 물어보면서 정리한다.
개념 이해 후 반드시 "그래서 Cursor 쓸 때 어떤 상황에 적용되는가"까지 연결한다.
연결이 안 되면 이해가 덜 된 것이다.

## Phase 2 — 팀원 대화 수집 (1주)

### 수집 방법

팀원 3~5명한테 요청한다.

요청 문구 예시:
"Cursor에서 AI한테 물어보다가 잘 안 풀려서 같은 질문 반복했던 대화 하나만 텍스트로 복사해서 줄 수 있어? 발표 자료 만드는 데 쓰려고. 익명으로 처리할게."

전체 내보내기 요청하면 부담스럽다. 딱 하나만 요청하는 게 받기 쉽다.

### 추가로 본인 데이터 수집

본인 Cursor 대화 1주일치를 직접 관찰한다.
- 어떤 질문에서 재질문이 발생했는가
- 컨텍스트를 처음부터 다시 설명한 적이 몇 번인가
- 코드/로그를 통째로 붙여넣고 "왜 그래요?" 한 적이 몇 번인가

### 수집 후 분석

모인 대화 텍스트를 Claude Code에 붙여넣고 패턴 분류한다.
찾을 패턴:

- 재질문 횟수 — 같은 맥락에서 추가 질문이 몇 번 발생했는가
- 컨텍스트 반복 — 매 질문마다 배경 설명을 처음부터 다시 썼는가
- 덩어리 질문 — 로그/코드를 통째로 붙여넣고 "이거 왜 그래요?" 유형
- 모호한 요청 — 구체적 조건 없이 "좋게 고쳐줘", "정리해줘" 유형
- 분할 질문 — 한 번에 물어볼 수 있는 내용을 여러 번에 나눠서 질문

3~5개만 모여도 공통 패턴 추출에 충분하다.

## 참고 사례 — dev-notes 학습 방식

지호님이 실제로 운영 중인 방식으로, 발표에서 "이런 활용도 가능하다"는 사례로 소개할 수 있다.

- 학습 내용을 마크다운 문서로 정리해서 로컬에 쌓는다
- AI와 대화할 때 관련 문서를 컨텍스트로 제공한다
- 결과적으로 같은 내용을 AI에게 반복 설명할 필요가 없어진다
- 문서가 쌓일수록 AI와의 대화 비용이 낮아지는 구조

Cursor에서도 동일한 원리를 적용할 수 있다. 설정 파일, 트러블슈팅 기록, 작업 가이드를 문서로 정리해두고 AI 대화 시 컨텍스트로 첨부하는 방식이다.

단 Claude Code 기반이라 그대로 따라하기보다 원리를 Cursor에 맞게 적용하는 방향으로 소개한다.

## 참고 사례 — Kiro IDE

AWS가 2025년 7월 출시한 AI 에이전트 IDE. 지호님이 수동으로 하던 방식을 IDE 레벨에서 구현한 것이다.

핵심 개념:

Steering Files — 프로젝트 전반에 AI가 항상 참고할 규칙을 마크다운으로 정의. 매번 채팅에서 반복 설명할 필요가 없어진다. CLAUDE.md, Cursor Rules와 동일한 원리.

Spec-Driven Development — 코드보다 명세(요구사항 → 설계 → 태스크)를 먼저 문서화하고, AI가 그 문서를 보고 일한다. 대화 맥락이 사라지는 문제를 구조화된 문서로 해결한다.

발표에서 활용 방향:
"내가 수동으로 하던 컨텍스트 구조화 방식을 AWS가 IDE로 만들었다 — 이게 단순한 팁이 아니라 업계가 검증한 방향이다"는 근거로 사용한다.

참고: kiro_ide.md

## 참고 사례 — AI-DLC (AI-Driven Development Lifecycle)

AWS가 2025년 re:Invent에서 발표한 개발 방법론. AI를 도구가 아니라 개발 프로세스 전체의 협업자로 재정의한다.

핵심 메시지: **컨텍스트의 연속성**

기존 방식은 AI를 각 단계에서 독립적인 도구로 쓴다. 요구사항은 사람이 분석하고 AI는 코드만 생성하는 구조라 AI가 전체 맥락을 모른다.

AI-DLC는 Inception → Construction → Operations 단계마다 컨텍스트가 누적된다. 요구사항 단계에서 논의한 내용이 설계에, 설계에서 결정한 내용이 운영까지 이어진다.

발표에서 활용 방향:
"AI 코딩 어시스턴트를 도입했는데 생산성이 오히려 떨어지는 팀이 많았다 — 도구 문제가 아니라 방식의 문제였다"는 AWS의 진단이 발표 도입부 문제 제기로 그대로 쓸 수 있다. 컨텍스트 없이 AI를 쓰는 것과 구조화된 컨텍스트를 가지고 쓰는 것의 차이를 업계 사례로 뒷받침한다.

참고: ai_dlc.md, kiro_ide.md

## Phase 3 — 전략 실험 (1주)

### 목표

Before/After 비교 데이터를 직접 만든다.
"이렇게 하면 다르다"를 주장하려면 본인이 먼저 검증해야 설득력이 있다.

### 실험 방법

실제 업무 태스크 3개를 선정한다.
트러블슈팅, 코드 리뷰, 문서 작성 중 각 하나씩.

각 태스크를 두 번 수행한다.
- 1회차: 기존에 하던 방식 그대로
- 2회차: Phase 1에서 이해한 원리를 적용한 방식

기록할 것:
- 질문 횟수 (몇 번 물어봤는가)
- 원하는 답이 나오기까지 걸린 시간
- 결과 품질 (주관적 평가 1~5점)

정밀한 측정보다 "눈에 보이는 차이"가 목적이다.
이 Before/After가 발표에서 가장 설득력 있는 파트가 된다.

## Phase 4 — 발표 자료 제작 (1주)

### 구성 (총 25~30분)

1부 — AI가 실제로 어떻게 동작하는가 (5~7분)
- 토큰, 컨텍스트 윈도우, 확률적 생성을 직관적으로 설명
- 복잡한 수식 없이, "그래서 Cursor 쓸 때 이런 상황이 생긴다"로 연결

2부 — 우리 팀에서 실제로 발생하는 낭비 패턴 (5~7분)
- Phase 2에서 수집한 실제 사례 기반
- 익명 처리, 패턴 이름 붙이기 ("덩어리 질문형", "컨텍스트 반복형" 등)

3부 — 즉시 적용 전략 5개 + Before/After (10분)
- Phase 3에서 실험한 결과 기반
- 각 전략마다 실제 Cursor 대화 캡처로 보여주기

4부 — 같이 해보기 (5분)
- 발표 자리에서 팀원이 직접 하나 해보는 것
- 준비한 예시 태스크를 나눠주고 두 가지 방식으로 물어보게 한다

### 타이틀 방향

"Cursor를 두 배로 잘 쓰는 법 — 우리 팀 대화 분석에서 뽑은 실전 전략"

범용 가이드가 아니라 우리 팀 데이터 기반임을 제목에서 드러낸다.

## 측정 — 평가 지표 근거 만들기

### 데이터 수집

팀원 3명에게 실제 대화 텍스트를 받는다.
"AI한테 잘 안 풀렸던 대화 하나만 복사해서 줘" — 부담 없이 요청할 수 있는 범위로 좁힌다.

민감한 서버 정보나 설정값이 포함될 수 있으므로 "민감한 값은 마스킹해서 줘"라고 사전 안내한다.

**도구별 추출 방법**

Cursor
- 대화창에서 전체 선택 후 복사 (1분 이내)

Claude Code
- `~/.claude/projects/` 경로에 프로젝트별 폴더로 저장됨
- 각 폴더 안 `.jsonl` 파일이 대화 기록
- 텍스트 편집기로 열거나 터미널에서 바로 확인 가능
- Claude Code는 회사 지원 도구가 아니므로 쓰는 팀원이 있을 때만 해당

### 분석

수집한 대화를 Claude에 붙여넣고 패턴 분류한다.
재질문 횟수, 컨텍스트 반복, 덩어리 질문, 모호한 요청 등 공통 패턴을 뽑아낸다.
이 결과가 발표의 "우리 팀 실제 사례" 파트가 된다.

### 설문 — 발표 효과 측정

대화 데이터는 발표 내용의 근거, 설문은 발표 후 실제 변화를 측정하는 용도다.

발표 전/후 동일하게 진행한다.

질문 예시:
- "AI에게 같은 내용을 반복해서 질문한 적이 얼마나 있었나요?"
- "AI 답변이 맥락을 못 잡아서 대화가 길어진 경험이 있었나요?"
- "컨텍스트(파일, 로그 등)를 먼저 제공하고 질문한 적이 있었나요?"

발표 2주 후 재실시, 변화 방향을 KPI 근거로 활용한다.
