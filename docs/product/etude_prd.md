## 1. 배경 및 문제 정의

### 문제

공공기관 납품 및 현장 작업 구조에서, 실제로 무엇을 어떻게 하는지 아는 인원이 소수에 집중되어 있다.
다른 구성원들은 현장에 나가도 작업 내용을 모르고, 장애가 발생해도 대응 방법을 모른다.

- 인수인계가 문서로만 이루어져 "읽었지만 못 한다"는 상황이 반복된다
- KodeKloud 같은 범용 실습 플랫폼은 존재하지만, 우리 회사 현장 상황을 반영한 커스텀 퀘스트를 만들 수 없다
- 현장 지식이 소수에게 집중된 채로 조직이 성장하면, 리스크가 점점 커진다

### 목적

특정 인원에게 쏠린 현장 지식을 조직 전체로 분산시킨다.
실무자가 현장 지식을 입력하면, 다른 구성원이 브라우저에서 직접 실습하며 숙지할 수 있는 구조를 만든다.

## 2. 목표 및 성공 지표

### MVP 목표

- 팀원이 브라우저에서 터미널을 열고 퀘스트를 풀 수 있다
- 채점이 자동으로 이루어진다
- 리눅스 / Docker / k8s 기초 세트를 실제로 풀 수 있다

### 성공 지표 (MVP)

- 팀원 중 1명 이상이 퀘스트를 처음부터 끝까지 완료
- "힌트 없이 지문만 보고 풀 수 있다"는 피드백
- 채점 오류 없이 정답/오답이 정확하게 판정됨

### 장기 목표

- 실무자가 퀘스트를 직접 등록하고, 다른 구성원이 그것으로 학습하는 선순환 구조
- 신규 입사자 온보딩에 실제로 사용되는 플랫폼
- 현장 작업 전 사전 실습 수단으로 정착

## 3. 사용자

### 트레이니 (학습자)

- 인프라/개발/운영 구성원
- 퀘스트 세트를 선택해 자율 학습하거나, 관리자가 할당한 세트를 완료
- 브라우저 터미널에서 직접 명령어를 실행하며 실습

### 실무자 (퀘스트 기여자, 플랫폼 안정화 이후)

- 현장 작업 경험이 있는 시니어 구성원
- 실제 작업/장애 사례를 텍스트로 입력 → AI가 퀘스트로 변환
- 생성된 퀘스트 최종 검토 및 승인

### 팀장 / 관리자

- 팀원 퀘스트 완료 현황 모니터링
- 세트를 팀원에게 할당하고 기한 설정

## 4. 기능 요구사항

### 4-1. MVP 범위 (현재 구현 중)

#### 브라우저 터미널 실습 환경

- 브라우저에서 터미널을 열고 명령어를 실행할 수 있다
- Docker 컨테이너 기반 샌드박스로 퀘스트마다 격리된 환경 제공
- xterm.js + WebSocket으로 실시간 입출력 스트리밍

#### 퀘스트 시스템

- 퀘스트 지문(미션 설명) + 힌트 + 채점 기준으로 구성
- 채점: 컨테이너 내부에서 grade_cmd 실행 → exit code로 성공/실패 판정
- 퀘스트 세트 단위로 묶어서 순서대로 진행

#### 퀘스트 세트 현황 (MVP)

| 세트 | 제목 | 환경 |
|------|------|------|
| 1 | 리눅스 기초 1 — 파일 탐색과 생성 | linux |
| 2 | 리눅스 기초 2 — 삭제·검색·권한 | linux |
| 3 | 리눅스 기초 3 — 프로세스와 시스템 | linux |
| 4 | 리눅스 네트워크/파일 전송 | linux-ssh |
| 5 | Docker 기초 | docker |
| 6 | k8s 기초 | k8s |
| 7 | 리눅스 압축과 아카이브 | linux |
| 8 | Docker 이미지 오프라인 반입 | docker-persistent |

#### 세트 선택 화면

- 카테고리(리눅스 / 도커 / k8s)별 아코디언 UI
- 세트 클릭 → 퀘스트 진행 화면으로 이동

#### Persistent Sandbox

- `docker save → load → tag → push` 같이 퀘스트 간 상태가 이어져야 하는 실습 지원
- 세트 종료 시 컨테이너 명시적 정리 (`/session/end`)

### 4-2. 다음 단계 (Phase 7~8)

#### 사용자 인증 + 진행 추적 (Phase 7)

- 로그인 (사내 이메일 기반, JWT)
- 퀘스트 완료 시 progress 기록
- 대시보드 — 내 완료 현황, 세트별 진행률
- 팀장 뷰 — 팀원 전체 완료 현황

#### 실배포 + 피드백 수집 (Phase 8)

- OCI Free Tier ARM VM에 배포 (Terraform + docker-compose)
- 화면 내 피드백 버튼 — 어느 화면에서든 피드백 제출 가능
- 제출된 피드백은 서버에 파일로 저장 (`feedback.jsonl`)

### 4-3. 장기 (플랫폼 안정화 이후)

#### AI 퀘스트 생성기

- 실무자 텍스트 입력 → Claude API → 퀘스트 자동 변환
- 생성된 퀘스트는 AI 에이전트 1차 리뷰 → 실무자 최종 승인 → 등록

#### 게임 모드 (Token Scrooge)

- 적은 AI 토큰으로 퀘스트를 완수한 사람이 이기는 경쟁 게임 모드
- 팀 행사/평가 자리에서 일반 트레이닝 대신 게임 모드로 전환
- AI 채팅 인터페이스 + 토큰 측정 + 실시간 리더보드 + 결과 리포트

#### 현장 실무 퀘스트 세트

- CMP 배포 세트 — 이미지 반입 → 배포 → 재시작 흐름
- k8s 현장 세트 — 롤링 업데이트, 롤백, 트러블슈팅
- CKA 준비 세트 — CKA 시험 빈출 영역 심화

## 5. 비기능 요구사항

### 보안

- 사내 서버 배포 (공공기관 관련 데이터 외부 유출 방지)
- 샌드박스 컨테이너는 퀘스트 종료 시 즉시 제거
- 로그인 없이는 퀘스트 접근 불가 (Phase 7 이후)

### 성능

- 컨테이너 생성 후 터미널 연결까지 5초 이내
- WebSocket 입출력 지연 체감 없는 수준

### 확장성

- 새 퀘스트 세트 추가 시 `init.sql` 수정만으로 반영 (코드 수정 없음)
- 새 샌드박스 타입 추가 시 `sandbox` 테이블에 행 추가만으로 반영

## 6. 범위 밖 (Out of Scope)

MVP 및 단기 계획에서 의도적으로 제외한 항목:

- AI 퀘스트 생성 — 플랫폼 안정화 후 가장 마지막 단계
- 게임 모드 (Token Scrooge) — 플랫폼 안정화 후 추가
- 퀘스트 관리 어드민 UI — 시점 미정
- 객관식/서술형 퀘스트 — 터미널 퀘스트 안정화 후 고려
- 외부 서비스 배포 (Vercel/Railway) — 사내 서버 우선
- 모바일 지원 — 터미널 특성상 데스크톱 전용

## 7. 시스템 구조

```
브라우저 (React + xterm.js)
    └── WebSocket / REST API → Fastify 서버

Fastify 서버 (Node.js + TypeScript)
    ├── WebSocket — 터미널 입출력 스트리밍
    ├── REST API — 퀘스트/세트/채점/피드백
    └── dockerode — Docker 컨테이너 제어

Docker 컨테이너 (샌드박스)
    ├── linux / linux-ssh / docker / docker-persistent / k8s
    └── 퀘스트마다 격리된 환경, 종료 시 제거

MariaDB
    └── sandbox / quest_set / quest / (user / quest_progress — Phase 7)
```
